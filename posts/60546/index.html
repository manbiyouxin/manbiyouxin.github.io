2、java有哪些锁？volatile、sync，lock的区别？Sync原理？volatile一定能保证线程安全吗？volatile原理？

synchronized vs lock

  • 语法层面:sync是关键字，源码在jvm层面，使用C++语言实现，lock是接口在jdk源码中，使用sync时，退出同步代码块时会自动释放锁，而lock需要自己释放锁，手动调用unlock方法

  • 功能层面:

    1. 二者都属于悲观锁，具有互斥、同步、可重入功能。同步：多个线程同时运行，当某个线程需要其他线程的结果，需要进行等待，只有拿到结果才可进行下一步的操作。sync可以使用wait，notify，lock可使用条件变量，await，signal来实现线程间的同步。（对已持有锁的对象重复加锁，加多道，解多道）。条件变量是当前的lock新建一个condition对象，一旦调用await方法则会当前owner中的线程放入对应的waiting queue中，signal则会按先入先出的顺序唤醒队列的中线程加入阻塞队列中，如果抢不到锁
    2. lock提供了sync所没有的功能，例如获取等待状态（可以知道其他哪些线程被阻塞了，同步，有哪些线程在等等）公平锁、可打断（不要无限制的等待下去）、可超时（设置超时时间）、多条件变量（多个等待队列）
    3. Lock有合适的应用场景的实现，reentrantLock，reentrantReadWriteLock

  • 性能层面

    1、在没有竞争时，sycn提供了很多的锁优化，例如偏向锁、轻量级锁，性能不赖

    2、竞争激烈时，lock的的性能更好

3、BIO、NIO、AIO区别，详细说一下BIO和NIO区别。多路复用？select、poll、epoll的区别。水平触发和边缘触发的区别。

  • IO模型的简单理解：用什么样的通道进行数据的发送和接受很大程度上决定了程序通讯的性能

  • Java的三种IO传输模型：BIO、NIO、AIO

    1. BIO:同步阻塞IO，服务器实现的模式为一个连接一个线程，即当客户端请求时，服务端为该请求启动一个线程进行处理，如果该连接不做任何事情则会造成不必要的开销。一般应用于连接少且固定的架构，并发局限于应用但编程设计较为简单

    2. NIO:(non-blocking IO)非阻塞同步IO，服务器会为多个连接注册一个多路复用器selector，通过不断轮询的方式去询问连接是否需要进行处理。它是面向块的或者是面向缓冲区编程的，它会将数据读取到稍后要使用到的缓冲区中，并且缓冲区中的数据可以前后移动，加大了数据的灵活性。当读取或写入时，当前数据用不到时，不会进行阻塞，线程会去做其他的事情。HTTP2.0用了多路复用的技术，同一个连接可以并发处理多个请求，并发请求的数量比http1.1大了好几个数量级。一般应用于连接时间短，连接数量多，操作级别轻的应用，例如聊天服务器，弹幕系统，服务间通讯等，设计比较复杂，jdk1.4后支持。

    3. AIO:JDK7引入了Asynchronous I/O，即AIO。在进行I/O编程中，常用到2种模式：Reactor反应器设计模式和Proactor。Java的NIO就是Reactor，当有事件触发时，会通知服务端做相应的处理。（类似多路复用选择器一直监听请求）。AIO即非阻塞异步IO，AIO引进异步通道，在操作系统先完成后再通知服务端程序启动线程去处理，一般应用于一些连接数量多，连接时间长的应用。例如相册服务器，充分调用os参与并发操作

HTTP 1.0和HTTP1.1 区别

  • 连接方法：HTTP 1.0为短连接，HTTP1.1使用了长连接，并通过管道机制进行数据传输，可以不用等待前一个请求响应结果，直接发送下一个请求。（但是一个线程也还是只能处理一个请求）

6.说一下线程池几个核心属性

  • corePoolSize核心线程数，线程池中最小可以运行的线程数

  • maximumPoolSize最大线程数,当队列中存放的任务达到队列容量时，当前可以运行的线程数变为最大线程数，即最小线程数=核心线程数+救急线程数。

  • keepAliveTime生存时间，救急线程的生存时间，生存时间内如果没有新任务，则会被释放

  • unit救急线程生存时间的单位

  • workQueue，任务队列，当没有空闲的核心线程时，新来的任务会加到任务队列，当任务队列也满时会执行救急线程来执行此任务

  • threadFactory线程工厂，用来定制线程的创建，例如线程的名称，是否为守护线程等。

  • handler 拒绝策列，当所有线程都繁忙时，任务队列也满时，会触发拒绝策列

    • aborPolicy，会直接抛出异常
    • discardPolicy，直接丢弃此任务，不抛异常
    • discardOldestPolicy，丢弃队列中中最早排队的任务，将其丢弃，入队
    • CallerRunsPolicy，由调用者线程来执行此任务

7.任务队列里有哪几种队列?

  • ArrayBlockingQueue:由数组构成的有界的阻塞队列，符合先进先出的原则。
  • LinkedBlockingQueue:由链表组成的有界阻塞队列（默认值是Integer.MAX_VALUE），吞吐量比ArrayBlockingQueue要好，静态工厂方法Executors.newFixedThreadPool用的就是这种队列
  • priorityBlockingQueue：支持优先级排序的无界阻塞队列
  • SynchronousQueue:不存放元素的阻塞队列，即单个元素的队列，每个线程的插入都要等到别的线程调用移除操作，否则插入操作一直处于阻塞状态。静态工厂方法Executors.newCachedThreadPool用的就是这种队列。吞吐量比LinkedBlockingQueue要好。
  • LinkedTransferQueue：由链表组成的无界阻塞队列。
  • LinkedBlockingDeque：由链表组成的双向阻塞队列。

8.尽可能说一下你了解的JVM知识，先说一下内存模型吧

•程序计数器是干嘛的

程序计数器是一块较小的内存，用来为字节码解释器指向对应的字节行，可以读取到对应的下一条指令，用来实现代码的流程控制，例如顺序执行、选择、循环、异常处理。

在多线程的情况下，程序计数器用来记录当前线程的位置，从而线程切换后可以恢复到正确的位置

•虚拟机栈是干嘛的？

虚拟机栈是JVM的核心区域，它随着线程的创建而创建随着线程的消亡而消亡，在线程中每调用一个方法都会在栈中压入一个栈帧，调用结束后会有一个栈帧被弹出符合先进后出原则。栈帧由三部分组成：局部变量表，操作数栈，动态链接。

  • 局部变量表：主要存放了编译期可知的数据类型（基本数据类型），以及对象引用（reference类型，它不同于对象本身，它主要是指向对象起始地址的一个指针或者是指向对象的句柄或者其他相关的位置）


  • 操作数栈：主要作为方法调用的中转站使用，用于存放方法执行过程中产生的中间计算结果。另外，计算过程中产生的临时变量也会放在操作数栈中。
  • 动态链接：主要服务一个方法需要调用其他方法的场景。在Java源文件编译成字节码文件时所有的变量和方法都转化为符号引用存放在常量池Class文件中。当一个方法要调用其他方法，需要将常量池中指向方法的符号引用转化为其在内存地址中的直接引用。实际上是对变量或者方法赋予了真实地址存放在运行时常量池中。动态链接的作用就是为了将符号引用转换为调用方法的直接引用。

•方法区是干嘛的？

方法区是JVM运行时数据区域的一块逻辑区域，是各个线程共享的内存区域。它是JVM规范自己所定义的一块区域由JVM本身自己去实现。当虚拟机要使用一个类时，它会加载解析类的class文件，将加载完的信息放在方法区中，包括类的信息，字段信息，方法信息，常量，即时编译器编译后的代码缓存等信息。

9.常见的HTTP的响应码，2xx是什么意思

1xx类属于提示信息，是协议处理中的一种中间状态，实际用到的很少

2xx类状态码表示服务器成功处理了客户端的请求，也是我们最愿意看到的状态。：

  • 200，最常见的状态码，表示一切正常，当请求为非head请求时，服务器返回的响应报文中携带响应体数据
  • 204，也是成功的响应码，但是返回的响应报文中没有响应体数据

3xx类状态码表示客户端请求的资源发生了变动，需要客户端用新的 URL 重新发送请求获取资源，也就是重定向。

  • 301，表示永久重定向，表示访问的资源已经不存在，需要用新的url再次访问

  • 302，表示临时重定向，说明请求的资源还在，但暂时需要用另一个 URL 来访问。

    301 和 302 都会在响应头里使用字段 Location，指明后续要跳转的 URL，浏览器会自动重定向新的 URL。

  • 304,不具有跳转的含义，表示资源未修改，重定向已存在的缓冲文件，也称缓存重定向，也就是告诉客户端可以继续使用缓存资源，用于缓存控制。

4xx类状态码表示客户端发送的报文有误，服务器无法处理，也就是错误码的含义。

  • 400表示客户端请求的报文有错误，但只是个笼统的错误。
  • 403 表示服务器禁止访问资源，并不是客户端的请求出错。
  • 404，表示请求的资源在服务器上不存在或未找到，所以无法提供给客户端。

5xx类 状态码表示客户端请求报文正确，但是服务器处理时内部发生了错误，属于服务器端的错误码.

  • 500，与 400 类型，是个笼统通用的错误码，服务器发生了什么错误，我们并不知道。
  • 501，表示客户端请求的功能还不支持，类似“即将开业，敬请期待”的意思
  • 502，bad gateway，通常是服务器作为网关或代理时返回的错误码，表示服务器自身工作正常，访问后端服务器发生了错误。
  • 503 表示服务器当前很忙，暂时无法响应客户端，类似“网络服务正忙，请稍后重试”的意思。

10.TCP粘包问题为什么是个误区

如何理解是 TCP 面向字节流协议？

之所以说TCP是面向字节流是协议，UDP是面向报文的协议，这是因为操作系统对TCP和UDP协议的发送机制不同，也就是问题在发送方。

UDP是面向报文的协议

当用户系统用UDP协议进行传输数据时，操作系统不会对消息进行拆分，在组装好UDP头部后就交给网络层去处理，所以发出去的UPD报文中的数据部分就是完整的用户信息，也就是每一个UDP报文就是用户消息的边界，这样接收方在接收到 UDP 报文后，读一个 UDP 报文就能读取到完整的用户消息。

如果收到了2个UDP的报文，系统是怎样区分的？操作系统收到报文后会先将报文插入到队列中，队列中的每一个元素就是一个UDP报文由用户调用系统函数，报文从队列中取出，然后从系统内核里拷贝给用户缓冲区

说说为什么 TCP 是面向字节流的协议？

当用户消息通过 TCP 协议传输时，消息可能会被操作系统分组成多个的 TCP 报文，也就是一个完整的用户消息被拆分成多个 TCP 报文进行传输。

这时，接收方的程序如果不知道发送方发送的消息的长度，也就是不知道消息的边界时，是无法读出一个有效的用户消息的，因为用户消息被拆分成多个 TCP 报文后，并不能像 UDP 那样，一个 UDP 报文就能代表一个完整的用户消息。

在发送端，当我们调用 send 函数完成数据“发送”以后，数据并没有被真正从网络上发送出去，只是从应用程序拷贝到了操作系统内核协议栈中。

至于什么时候真正被发送，取决于发送窗口、拥塞窗口以及当前发送缓冲区的大小等条件。也就是说，我们不能认为每次 send 调用发送的数据，都会作为一个整体完整地消息被发送出去。

因此，我们不能认为一个用户消息对应一个 TCP 报文，正因为这样，所以 TCP 是面向字节流的协议。

当两个消息的某个部分内容被分到同一个 TCP 报文时，就是我们常说的 TCP 粘包问题，这时接收方不知道消息的边界的话，是无法读出有效的消息。

要解决这个问题，要交给应用程序。

如何解决粘包问题

  1. 固定长度的消息：这种是最简单方法，即每个用户消息都是固定长度的，比如规定一个消息的长度是 64 个字节，当接收方接满 64 个字节，就认为这个内容是一个完整且有效的消息。但是这种方式灵活性不高，实际中很少用。

  2. 特殊字符作为边界：我们可以在两个用户消息之间插入一个特殊的字符串，这样接收方在接收数据时，读到了这个特殊字符，就把认为已经读完一个完整的消息。

    HTTP 通过设置回车符、换行符作为 HTTP 报文协议的边界。

    有一点要注意，这个作为边界点的特殊字符，如果刚好消息内容里有这个特殊字符，我们要对这个字符转义，避免被接收方当作消息的边界点而解析到无效的数据。

  3. 自定义消息结构：我们可以自定义一个消息结构，由包头和数据组成，其中包头包是固定大小的，而且包头里有一个字段来说明紧随其后的数据有多大。

    比如这个消息结构体，首先 4 个字节大小的变量来表示数据长度，真正的数据则在后面。

    struct { 
    u_int32_t message_length; 
    char message_data[]; 
} message;

    当接收方接收到包头的大小（比如 4 个字节）后，就解析包头的内容，于是就可以知道数据的长度，然后接下来就继续读取数据，直到读满数据的长度，就可以组装成一个完整的用户消息来处理了。

什么是微服务

微服务是一种架构模式或者说是架构风格，它提倡将单一的应用程序划分成一个一个的小服务，每个服务运行在其自己的进程中，服务之间相互调用，相互协同，为用户提供最终的价值。服务之间采用轻量级的通讯机制相互沟通（通常是基于HTTP的Restful API），每个服务都围绕具体的业务进行构建，可以独立的部署维护在测试环境，生产环境等环境中。另外，应避免统一的、集中式的项目管理，对具体的一个服务而言，应根据业务上下文，选择合适的语言、工具对其进行构建，可以有一个非常轻量级的集中式管理来协调这些服务，可以使用不同的语言来编写服务，也可以使用不同的数据存储。

简而言之：拒绝大型单体应用，基于业务边界进行服务微化拆分，各个服务独立部署运行

  2. 缺点

什么是网络分区？：分布式系统中，多个节点之前的网络本来是连通的，但是因为某些故障（比如部分节点网络出了问题）某些节点之间不连通了，整个网络就分成了几块区域，这就叫 网络分区。

RabbitMQ

是AMQP（Advanced Message Queue Protocol）高级消息队列协议（消息代理的规范）的一种实现。

核心概念
  • Message：消息是不具名的。由消息头和消息体构成，消息体是不透明的，消息头由一系列可配置参数构成，例如routing-key路由键，priority消息的优先级，是否可持久化保存
  • Publisher：消息的生产者，也是将消息发送给交换器的应用程序
  • Exchange：交换器，用来接受生产者发布的信息并将信息路由给服务器中的消息队列，有direct、fanout、topic、headers，不同Exchange的转发策略有所不同。
  • Queue:消息队列，用来保存消息直到发送给消费者，是消息的容器，也是消息的终点。消息可以投入一个或多个队列中，消息一直在消息队列中，直到消费者连接消息队列将其取走
  • Binding:绑定，用于消息队列和交换器之间的关联。一个绑定就是基于路由键将交换器和消息队列绑定起来的规则，所以可以将交换器理解为由绑定构成的路由表
  • Connection：网络连接，比如一个TCP连接
  • Channel：信道，多路复用连接中的一条独立的双向数据流通道。信道是建立在真实的TCP连接内的虚拟连接，AMQP 命令都是通过信道发出去的，不管是发布消息、订阅队列还是接收消息，这些动作都是通过信道完成。因为对于操作系统来说建立和销毁 TCP 都是非常昂贵的开销，所以引入了信道的概念，以复用一条 TCP 连接。
  • Consumer：消息的消费者，表示一个从消息队列中取得消息的客户端应用程序
  • Broker：表示消息队列服务器实体
消息路由

生产者把消息发布到Exchange上，Binding决定消息发送给哪个队列，消息最终到达消息队列由消费者接受

Exchange类型：根据消息分发的不同策列共分为4中类型。其中headers有header直接确定队列，性能差很多，一般不用了

  • direct，

  • 消息中的路由键（routing key）如果和Binding 中的 binding key 一致， 交换器
    就将消息发到对应的队列中。路由键与队列名完全匹配，如果一个队列绑定到交换机要求路由键为“dog”，则只转发 routingkey 标记为“dog”的消息，不会转发“dog.puppy”，也不会转发“dog.guard”等等。它是完全匹配、单播的模式

  • fanout，扇出，每个发到 fanout 类型交换器的消息都会分到所有绑定的队列上去。fanout 交换器不处理路由键，只是简单的将队列绑定到交换器上，每个发送到交换器的消息都会被转发到与该交换器绑定的所有队列上。很像子网广播，每台子网内的主机都获得了一份复制的消息。fanout 类型转发消息是最快的

  • topic，交换器通过模式匹配分配消息的路由键属性，将路由键和某个模式进行匹配，此时队列需要绑定到一个模式上。它将路由键和绑定键的字符串切分成单词，这些单词之间用点隔开。它同样也会识别两个通配符：符号“#”和符号“”。#匹配0个或多个单词，匹配一个单词

消息的可靠到达
  1. 消息只要被broker接受到就会发送comfirmCallback，但是只能保证消息到达了服务器，并不能保证消息加入了队列中，所以有了接下来的returnCallback回调机制。如果未能投递到queue中则会记录详细的数据，定期的巡查或纠错都需要这些数据。
  2. 消息从消息队列到达消费者会自动调用ACK回调，并从broker中的queue中移除，也可以手动回调，选择接受或者不接受（NoACK），可用channel.baseReject(message.getMessageProperties().getDeliveryTag(),false)false来拒收且不再放回消息队列中。
